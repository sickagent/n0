# ADR-001: Архитектура AI-BI платформы «n0»

**Статус:** Принято  
**Дата:** 2026-04-16  
**Автор:** Platform Team  
**Область:** Backend, Data Platform, AI Integration  

---

## 1. Контекст и ограничения

Требуется построить корпоративную BI-платформу нового поколения, в которой аналитическая логика, генерация SQL и построение визуализаций делегированы внешним ИИ-агентам. Система выступает в роли **исполнительного и мета-слоя**:
- Предоставляет агентам структурированный контекст о доступных данных (схемы, датасеты, семантический слой).
- Обеспечивает безопасный, изолированный и аудируемый доступ к источникам данных.
- Исполняет запросы, кэширует результаты и возвращает данные в формате, готовом для рендеринга графиков и дашбордов.

**Ключевые ограничения:**
- Бэкенд на **Go**.
- Метаданные в **PostgreSQL**.
- Внутренняя коммуникация сервисов через **NATS** (Core + JetStream + KV).
- Мультиарендность, горизонтальная масштабируемость и отказоустойчивость.

---

## 2. Архитектурные принципы

1. **Система — руки агента, не его мозг.**  
   Агент решает *что* спросить, система гарантирует *как безопасно и эффективно* выполнить.

2. **Все аналитические запросы — асинхронные jobs.**  
   Любой запрос создаёт долгоживущую задачу с `job_id`. Агент получает результат через polling или webhook.

3. **Stateless сервисы.**  
   Локальное состояние запрещено. Кэш, сессии, очереди и координация — во внешних системах (Redis, NATS, PostgreSQL, S3).

4. **Zero-trust к агентам.**  
   Агент аутентифицируется, работает в рамках RBAC, не имеет прямого доступа к БД. Все SQL-запросы проходят sandbox-валидацию.

## 4. Протокольный стек

### 4.1. Внешний интерфейс (Агент → Система)

| Протокол | Назначение | Когда использовать |
|----------|------------|-------------------|
| **MCP** (Model Context Protocol) | Интеграция с Claude Desktop, Cursor, IDE-агентами | Экспериментальные и user-facing интеграции |
| **gRPC over HTTP/2** | Корпоративные агенты, чат-боты, embedded analytics | Production, где критична производительность и типизация |
| **REST** | Фронтенд дашбордов, внешние интеграции | Простота интеграции для сторонних систем |

**Agent Gateway** реализует MCP Server и gRPC/REST Gateway как отдельные процессы, но оба работают с одним и тем же внутренним API.

### 4.2. Внутренняя коммуникация (Сервис → Сервис)

Все внутренние сервисы общаются через **NATS**:
- **NATS JetStream** — надёжные очереди и персистентные стримы.
- **NATS Core** — лёгкий pub/sub и request-reply для синхронных вызовов.
- **NATS KV** — распределённое key-value хранилище для координации.

**Почему NATS:**
- Единая технология для messaging, RPC и coordination.
- Встроенная горизонтальная масштабируемость.
- Минимальный operational overhead по сравнению с Kafka + gRPC mesh + etcd.
- Отличная поддержка в Go (`nats.go` + `micro` для NATS-based services).

## 5. Потоки данных

### 5.1. Discovery Flow (агент узнаёт структуру данных)

```
1. Агент → Agent Gateway: MCP tool call / gRPC GetSchema()
2. Agent Gateway: аутентификация + rate limiting
3. Agent Gateway ──[NATS Request-Reply]──► Meta Service
4. Meta Service: читает SchemaSnapshot из PostgreSQL
5. Meta Service ──[NATS Reply]──► Agent Gateway
6. Agent Gateway → Агент: JSON схема с типами колонок, таблицами, relations
```

### 5.2. Execution Flow (агент отправляет запрос)

```
1. Агент → Agent Gateway: SubmitQuery(SQL or structured query)
2. Agent Gateway ──[NATS Request-Reply]──► Meta Service: проверка прав на dataset/connection
3. Agent Gateway ──[NATS JetStream: PUB to queries.pending]──► Query Engine
4. Query Engine: возвращает job_id
5. Agent Gateway → Агент: job_id + estimated_time

[Асинхронное выполнение]
6. Query Engine Worker: забирает job из JetStream consumer group
7. Query Engine: Sandbox-валидация SQL, проверка кэша Redis
8. Query Engine ──[NATS Request-Reply]──► Connection Manager: получить пул + credentials
9. Worker: исполняет SQL в Source DB с context.WithTimeout()
10. Worker: сохраняет результат в S3 (большие) или Redis (малые)
11. Query Engine ──[NATS Core: PUB to events.query.completed]──► уведомление агенту
12. Агент: GET /jobs/{job_id}/result → получает данные
```

### 5.3. Visualization Flow

```
1. Агент получает данные и вызывает SuggestChartConfig(data_schema, intent)
2. Agent Gateway ──[NATS Request-Reply]──► Query Engine (Result Formatter)
3. Formatter анализирует типы колонок, предлагает тип графика
4. Агент получает: { data: [...], chart_config: { type: "line", x: "date", y: "revenue" } }
5. Агент/Клиент рендерит график локально через ECharts/Plotly/Vega
```

## 6. Использование NATS

### 6.1. NATS JetStream — Очереди запросов

| Stream | Subject | Назначение |
|--------|---------|------------|
| `QUERIES` | `queries.pending.{tenant_id}` | Новые аналитические запросы |
| `QUERIES` | `queries.retry.{tenant_id}` | Повторные попытки после временной ошибки |
| `AUDIT` | `audit.events.{tenant_id}` | Асинхронный поток аудит-событий |

**Consumer Group:** `query-engine-workers`  
**Политика доставки:** Exactly-once (дедупликация по `job_id`)  
**Ack Policy:** Explicit (воркер ack'ает только после успешного сохранения результата)  
**Max Deliveries:** 3, затем DLQ (`queries.dead_letter`)

### 6.2. NATS Core — Event Bus

| Subject | Publisher | Subscribers | Назначение |
|---------|-----------|-------------|------------|
| `events.schema.updated` | Meta Service | Query Engine, Redis Cache | Инвалидация кэша схем |
| `events.query.completed` | Query Engine | Audit Logger, Webhook Dispatcher | Аудит и уведомления |
| `events.credentials.rotated` | Connection Manager | Query Engine Workers | Обновление пулов |
| `events.tenant.quota.exceeded` | Query Engine | Rate Limiter, Alerting | Управление квотами |

### 6.3. NATS Request-Reply — Синхронный RPC

| Subject | Requester | Responder | Таймаут |
|---------|-----------|-----------|---------|
| `svc.meta.get_schema` | Agent Gateway | Meta Service | 2s |
| `svc.meta.check_permissions` | Agent Gateway / Query Engine | Meta Service | 1s |
| `svc.conn.get_pool` | Query Engine | Connection Manager | 500ms |
| `svc.conn.health_check` | Query Engine | Connection Manager | 1s |
| `svc.query.get_status` | Agent Gateway | Query Engine | 1s |
| `svc.query.suggest_chart` | Agent Gateway | Query Engine | 2s |

### 6.4. NATS KV — Координация и состояние

| Bucket | Назначение | TTL |
|--------|------------|-----|
| `tenant_quotas` | Оставшиеся квоты на запросы в минуту/час | 1h |
| `circuit_breakers` | Состояние CB для каждого connection_id | 5m |
| `active_pools` | Registry: какой worker держит пул к какой БД | 30s |
| `gateway_sessions` | MCP session state для агентов | 24h |

## 7. Отказоустойчивость и масштабирование

### 7.1. Горизонтальное масштабирование

| Компонент | Стратегия |
|-----------|-----------|
| **Agent Gateway** | Kubernetes HPA по RPS. Stateless, любая реплика может обработать любой запрос. |
| **Meta Service** | HPA по CPU. PostgreSQL read replicas для тяжёлых catalog search запросов. |
| **Query Engine** | HPA по длине JetStream consumer pending messages. Workers масштабируются автоматически. |
| **Connection Manager** | HPA + sharding по `connection_id` (consistent hashing) для избежания thundering herd. |
| **NATS Cluster** | 3+ nodes в Raft-группе. JetStream реплицирует стримы настроенным количеством реплик. |

### 7.2. Устойчивость к отказам

| Сценарий | Защита |
|----------|--------|
| Worker упал во время выполнения запроса | JetStream redelivers job другому worker после timeout (NAK или lack of ack) |
| Source DB недоступна | Connection Manager Circuit Breaker быстро отклоняет запросы, пока health-check не восстановится |
| Тяжёлый запрос «завис» | `context.WithTimeout()` прокидывается до драйвера БД; job помечается failed |
| Redis недоступен | Query Engine продолжает работу, но без кэша. Meta Service читает напрямую из PostgreSQL. |
| NATS node упал | Кластер NATS автоматически перевыбирает лидера для JetStream; клиенты реконнектятся. |

### 7.3. Graceful Degradation

- Если Source DB read replica доступна, Connection Manager может переключить трафик на неё для SELECT-запросов.
- Если Meta Service перегружен, Agent Gateway может вернуть агенту stale схему из локального кэша (с TTL 5 минут).
- Для MCP-интеграций предусмотрен fallback на REST API при недоступности gRPC.

## 8. Безопасность

### 8.1. Tenant Isolation
- Все таблицы PostgreSQL для метаданных имеют `tenant_id` + Row-Level Security (RLS) policies.
- NATS subjects всегда включают `{tenant_id}` или сегментируются через NATS Accounts.
- Кэш в Redis ключируется с префиксом `tenant:{id}:`.

### 8.2. Credential Security
- Пароли и API-ключи никогда не хранятся в PostgreSQL метаданных.
- Connection Manager запрашивает секреты runtime из **HashiCorp Vault** с lease renewal.
- Все connection strings строятся с read-only ролями на уровне целевой БД.

### 8.3. Query Sandbox
Перед исполнением SQL проходит валидацию:
- **Whitelist tables:** запрос может обращаться только к таблицам, явно разрешённым в датасете.
- **Blacklist commands:** запрещены DDL (`CREATE`, `DROP`, `ALTER`) и DML (`INSERT`, `UPDATE`, `DELETE`).
- **Timeout:** hard limit на исполнение (например, 60 секунд).
- **Row limit:** `LIMIT` инжектируется автоматически, если агент его не указал (default 10 000).
- **RLS injection:** для каждого запроса автоматически добавляются фильтры по tenant-политикам.

### 8.4. Аудит
- Все действия агентов публикуются в NATS Stream `AUDIT`.
- Отдельный сервис Audit Logger читает стрим и пишет в **ClickHouse** или отдельный шард PostgreSQL.
- Лог содержит: `tenant_id`, `agent_id`, `action`, `query_text`, `connection_id`, `execution_time_ms`, `row_count`, `success/failure`.

## 9. Альтернативы и обоснование выбора

### Почему не адаптированный datalens-backend?
DataLens требует обязательного Node.js-сервиса United Storage (US), жёстко связан с UI-ориентированными абстракциями (`BlockLegend`, `QuerySpec`) и не имеет точек расширения для LLM-агентов. Переиспользование потребовало бы 4–6 месяцев реверс-инжиниринга.

### Почему PostgreSQL для метаданных, а не MongoDB?
Модель BI-метаданных глубоко реляционна (`workspace → connection → schema → table → column → dataset → dashboard`). PostgreSQL с JSONB даёт необходимую гибкость без потери ACID, foreign keys и мощных возможностей поиска (`pg_trgm`, `btree_gin`).

### Почему NATS вместо Kafka + gRPC mesh?
- Единая технология для messaging, RPC и coordination.
- Гораздо меньший operational footprint.
- Встроенная поддержка request-reply и KV store.
- Отличная интеграция с Go-экосистемой.

## 10. Расширяемость: Плагины и Адаптеры

Система спроектирована как **платформа**, а не монолит. Пользователи и команды могут расширять функциональность без изменения core-кода. Архитектура плагинов вдохновлена подходом DataLens (разделение ответственности на слои подключения, схемы и исполнения), но адаптирована под Go-экосистему и корпоративные требования к изоляции и языковой независимости.

### 10.1. Философия плагинов

В DataLens каждый коннектор — это три пересекающихся Python-пакета (`dl_core`, `dl_api_lib`, `dl_formula`), регистрируемые через `entrypoints`. В n0 мы сохраняем ту же **трёхслойную логику**, но упрощаем интеграцию:

| DataLens (Python) | n0 (Go + gRPC/NATS) | Смысл |
|-------------------|------------------------------|-------|
| `dl_core` — подключение, пулы, адаптеры | **Connection Adapter** | Как подключиться и выполнить запрос |
| `dl_api_lib` — схемы, формы, валидация | **Schema Provider** | Как описать структуру данных для агента |
| `dl_formula` — диалект, функции, SQL | **Dialect Engine** | Как генерировать и валидировать SQL для конкретной БД |

Вместо Python `entrypoints` используется **Plugin Registry** в Meta Service + **gRPC-протокол** для внешних плагинов. Это позволяет писать плагины на **любом языке** (Go, Python, Rust, Java), запускать их в изолированных контейнерах и версионировать независимо от core.

### 10.2. Типы плагинов

Система поддерживает четыре класса плагинов:

1. **Database Adapter Plugin** — подключение к новым источникам данных (SQL и NoSQL).
2. **Agent Capability Plugin** — расширение набора tools, доступных агенту через MCP/gRPC.
3. **Query Transformer Plugin** — перехватчики (hooks) на этапах жизненного цикла запроса.
4. **Formatter & Visualization Plugin** — новые форматы вывода данных и типы чартов.

### 10.3. Database Adapter Plugin

**Назначение:** добавить поддержку новой БД (корпоративный Data Warehouse, специфичный API, внутренний кластер).

**Протокол взаимодействия:** gRPC-контракт `DatabaseAdapter` с обязательными методами:

| Метод | Что делает | Кто вызывает |
|-------|------------|--------------|
| `GetAdapterInfo()` | Возвращает название, версию, автора, поддерживаемые возможности | Plugin Registry при регистрации |
| `TestConnection(params)` | Проверяет доступность источника с given credentials | Connection Manager, Meta Service |
| `GetSchema(params, options)` | Возвращает структуру базы (таблицы, колонки, типы, индексы) | Meta Service (фоновое обновление снапшота) |
| `ExecuteQuery(params, query, options)` | Выполняет SQL или нативный запрос и возвращает поток/батчи строк | Query Engine Worker |
| `GetDialectCapabilities()` | Сообщает поддерживаемые SQL-функции, типы данных, синтаксические ограничения | Query Engine (Sandbox validator) |
| `CompileQuery(abstract_query)` | *Опционально.* Преобразует структурированный запрос системы в SQL целевой БД | Query Engine (для сложных диалектов) |

**Формы поставки адаптера:**

| Тип | Когда использовать | Изоляция |
|-----|-------------------|----------|
| **Built-in** | Стандартные адаптеры (PostgreSQL, ClickHouse, MySQL), поставляемые с core | В том же процессе Connection Manager |
| **External gRPC** | Пользовательские адаптеры, корпоративные интеграции | Отдельный pod/sidecar, общается по gRPC |
| **WASM** | Лёгкие адаптеры с минимальной логикой (например, обёртка над ODBC) | Загружается в sandbox-рuntimе Connection Manager |

**Flow добавления нового адаптера:**
1. Пользователь разворачивает gRPC-сервис с имплементацией `DatabaseAdapter`.
2. Через API Meta Service вызывает `RegisterPlugin(type=DB_ADAPTER, endpoint=grpc://...)`, указывая `tenant_id` (плагин может быть глобальным или tenant-scoped).
3. Plugin Registry сохраняет метаданные плагина и шлёт `events.plugin.registered` в NATS.
4. Connection Manager подписывается на событие, добавляет endpoint в пул доступных адаптеров.
5. При создании connection администратор видит новый тип БД в выпадающем списке.
6. Meta Service периодически вызывает `GetSchema()` через Connection Manager и кэширует снапшот.

**Query Engine → Connection Manager → Adapter:**
- Query Engine не знает о существовании плагинов напрямую.
- Он просит Connection Manager: `ExecuteQuery(connection_id, sql)`.
- Connection Manager определяет по `connection_id`, какой адаптер (built-in или external) должен обработать запрос, и проксирует вызов.

### 10.4. Agent Capability Plugin

**Назначение:** добавить агенту новые инструменты (tools) помимо стандартных `query_data` и `get_schema`.

**Примеры capabilities:**
- `explain_dataset` — агент может попросить плагин объяснить бизнес-логику конкретного датасета.
- `forecast` — плагин оборачивает внешний ML-сервис, агент вызывает его для прогнозирования временных рядов.
- `alert_configurator` — плагин помогает агенту создавать алерты на основе данных.

**Протокол:**
- Плагин регистрирует список `capabilities` (название tool, JSON-schema входных/выходных параметров, описание для LLM).
- Agent Gateway динамически агрегирует capabilities всех активных плагинов и включает их в `tools` секцию MCP- или gRPC-контракта при сессии с агентом.
- При вызове tool Agent Gateway делает `NATS Request-Reply` (или gRPC) к плагину.

**Жизненный цикл:**
1. Плагин регистрируется в Plugin Registry.
2. Agent Gateway читает реестр при старте и по событию `events.plugin.registered`.
3. При подключении агента Gateway формирует полный список доступных tools = встроенные + tenant-specific плагины.

### 10.5. Query Transformer Plugin

**Назначение:** модифицировать или обогащать запросы на разных этапах без изменения core.

**Hook-этапы:**

| Hook | Когда срабатывает | Пример использования |
|------|-------------------|----------------------|
| `HOOK_VALIDATE` | После sandbox-валидации, перед планированием | Корпоративный compliance-плагин запрещает запросы к определённым таблицам |
| `HOOK_PLAN` | Перед отправкой в Source DB | Плагин добавляет query hints для конкретной БД или rewrite для оптимизации |
| `HOOK_EXECUTE` | После получения результата, перед кэшированием | Плагин анонимизирует PII-данные на лету |
| `HOOK_FORMAT` | Перед отдачей агенту | Плагин преобразует единицы измерения (валюта, метрики) |

**Реализация:**
- Через **NATS Core pub/sub** на темы `hooks.{stage}.{tenant_id}`.
- Или через **WASM** для lightweight-логики (например, маскирование телефонов/емейлов).
- Query Engine выполняет цепочку плагинов в порядке приоритета, указанного в Plugin Registry.

### 10.6. Plugin Registry и жизненный цикл

**Хранение:** Meta Service (PostgreSQL).

**Основные сущности:**
- `plugin_definitions` — метаданные плагина (id, type, name, version, author, endpoint, protocol).
- `plugin_capabilities` — декларативные возможности (список tools, поддерживаемые БД, hook-этапы).
- `tenant_plugins` — какие плагины активны для какого tenant'а.
- `plugin_healthchecks` — последний heartbeat, статус (healthy/unhealthy).

**Жизненный цикл плагина:**

```
REGISTERED → VALIDATED → ACTIVE ↔ DEGRADED → DEPRECATED → REVOKED
```

- **Registered:** пользователь загрузил манифест.
- **Validated:** Plugin Registry проверил доступность endpoint (gRPC health-check или NATS heartbeat).
- **Active:** плагин используется в production.
- **Degraded:** heartbeat потерян, плагин исключается из routing до восстановления.
- **Deprecated:** администратор пометил плагин устаревшим, новые connection'ы не создаются.
- **Revoked:** полное удаление, все связи с tenant'ами разорваны.

**Discovery механизм:**
- Внешние плагины шлют **heartbeat** в NATS KV bucket `plugin_healthchecks` каждые 30 секунд.
- Plugin Registry периодически сканирует KV и обновляет статус.
- При появлении нового healthy-плагина рассылается `events.plugin.registered`.

### 10.7. Безопасность плагинов

- **Tenant scope:** по умолчанию плагин доступен только tenant'у, который его зарегистрировал. Глобальные плагины требуют роли `platform.admin`.
- **Network isolation:** external gRPC-плагины запускаются в отдельном Kubernetes namespace с NetworkPolicy, ограничивающей egress только к разрешённым БД.
- **Credential proxying:** плагин никогда не получает plaintext-пароли. Connection Manager выступает proxy и подменяет credentials на short-lived токены.
- **Resource quotas:** для external плагинов настраиваются CPU/memory limits и timeout на все gRPC-вызовы.
- **Audit:** все вызовы плагинов (`ExecuteQuery`, `RunTool`, `Hook`) логируются в AUDIT-стрим.
